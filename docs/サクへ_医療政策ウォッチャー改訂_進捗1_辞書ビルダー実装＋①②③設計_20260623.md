## ✅ 完了：④ VOICEPEAK辞書ビルダー（`src/voicepeak_dict_builder.py`・新規）

**狙い（依頼の本丸）**：1本化で総量が増えても、緑さんの「読み間違い修正」を増やさない。原稿から固有名詞を拾い、`src/podcast_dictionary.json`（既存144件・`[{"sur","pron"}]`）へ半自動で蓄積。

**実装の要点**
- **オフライン候補抽出（鍵不要）**：英字略語/頭字語（WHO・NDB・FIP）、英数字混在（G20・SDGs・COVID-19）、数字+単位（2日間・5疾病）、連続漢字（難読源）を正規表現で抽出し、**辞書未登録のものだけ**を候補化。
- **Gemini読み付与（任意・--llm）**：候補＋原稿文脈を渡し、「誤読しやすく読みを固定すべき語だけ」を `{"sur","pron(カタカナ)}` で出力（既存 `github_content_generator.py` と同一SDK `google-genai`・同一 config ローダ・temperature 0.1）。応答は ```json フェンスや前後文を許容して頑健にパース。
- **🔴 既存の人手読みは絶対に上書きしない**：マージは `sur` 重複時に既存 `pron` を温存（追加 or スキップのみ）。緑さん/木内さんが手で直した読みを壊さない（A患者調査の秘匿NULL・薬局func_*のunknownと同じ faithfulness ガードの思想）。
- **保存形式を維持**（`ensure_ascii=False, indent=2`）。新規は末尾追記で差分が読みやすい。
- **既存パイプライン非破壊**：`podcast_dictionary.json` は現状どのコードからも未参照の孤立ファイル。本モジュールも独立。台本生成直後フックは**自動配線せず**、本書の提案を承認後に組み込む。

**検証（オフライン・このリポジトリで実行済）**
- 144件ロード → サンプル原稿から候補抽出（WHO/SDGs/G20/2024年は既存辞書にあり正しく除外、中医協など未登録は候補化）。
- マージ：新規末尾追記・既存 `日本=ニホン` を `ニッポン` で上書きしようとしても温存・`sur` dedup・JSON往復で保存形式維持、をアサートで確認。
- CLI `--no-llm`：候補を `*_candidates.txt`（`表層\t（読み空欄）`）で出力＝人手 or 後でLLM。
- ※Gemini読み生成は鍵が無く未実行（コードは既存稼働パスと同型）。本番で1回 `--dry-run` 確認を推奨。

**使い方**
```

## 📋 実装設計：①②③（要サクレビュー／本体パイプライン変更）

調査で確定したNotionプロパティ・状態遷移・生成フローに基づく。`github_content_generator.py`（CI現行）に集約し、ローカル系 `notion_content_generator.py` は今回触らない。

### ① Notion『優先度』プロパティ（高/低）を追加して読む
- **Notion DB**：`Priority`（select、値「高」「低」）を新設＝**木内さんが選定時に付与**（コード外の手作業）。優先度（刺さり度）と国内/国際は**独立2軸**。国内/国際は既存の**URLドメイン判定**（`weekly_policy_report.py` `SOURCE_TAG_MAP` L46 / `is_international()` L477）を流用＝Notionに国際フラグは増やさない。
- **読み取り**：`github_content_generator.py` の `NotionAPI.get_property()` L543 で `Priority` を取得。`process_pdf_pages()` L1154 / `process_url_pages()` L1223 で各ページの優先度を保持。未設定時の既定は「低」（安全側＝厚くしすぎない）。

### ② 生成を優先度で分岐（個別ニュースは現状の約2/3量）
- `generate_from_pdfs()` L870 / `generate_from_url()` L987 に優先度引数を追加。
  - **高**：従来の記事＋（含意つき）厚め。ただし全体目標尺のため本文は現状比やや圧縮。
  - **低**：新規プロンプト `PROMPT_BLOG_BRIEF`（`PROMPT_BLOG_PDF` L109 の隣に追加）＝**一行ヘッドライン＋1〜2文**。
- FC待ちへの遷移（L1210 で `ファクトチェック待ち`）は不変＝**FCは木内さんが手動**（依頼の「変えない3」を厳守）。
- ⚠️**横展開の注意**：CI系のステータス文字列は小文字 `執筆待ち(pdf)/(url)`（L1159/L1228）、automation/ローカル系は大文字 `執筆待ち(PDF)/(URL)`。分岐実装時に表記を統一（揺れがバグ源）。

### ③ FC完了後＝その日の全ニュースを1記事・1音声に統合
- **新規 `src/daily_digest_generator.py`**（稼働中の `weekly_policy_report.py` を雛形に＝実績あるパターン）：
  - 当日FC完了分を抽出：`weekly_policy_report.query_completed_articles(days=1)` L242 を「当日 `Date(W-complete)` or FC完了」基準に。
  - 各記事の `Article(Web)` 子ページ本文取得＝`get_page_text()` L301 流用。
  - **濃淡**：`Priority==高`＝含意つきで厚く／`Priority==低`＝一行。**国際保健（低含む）は毎日必ず一行**（`is_international()` の記事は最低一行を保証＝差別化・溜めない）。
  - **目標尺の制御**：1音声 **12〜15分** を先に決め、高の「厚さ×本数」で調整・低は一行で総量を暴れさせない（ローカル系 `PROMPT_PODCAST` L125 に12〜15分の前例あり）。
  - 統合本文を1本に整形（`format_weekly_report()` L626 の丸数字列挙・国内→国際ソート L858 を日次へ改変）→ 統合台本は `PROMPT_SCRIPT` L253 を**1回だけ**適用（句点改行・プレーン・冒頭挨拶禁止の既存ルール踏襲）→ Notion子ページ保存（`save_to_notion()` L1079 流用）。
  - 統合台本確定後に **④の `update_dictionary_from_script()` を呼ぶ**（辞書整備をパイプラインに組み込む＝本丸の自動化）。
- **新規 workflow** `.github/workflows/daily_digest.yml`（`weekly_policy_report.yml` 雛形・cron は毎日夕方＝FCが揃う時間）。
- **二重生成の回避**（設計の肝）：②=「素材（個別記事/台本）」生成、③=「当日素材を1本に統合」。統合は素材の本文を**束ねるだけ**で個別ニュースを再生成しない。個別音声化（現 `Status(Podcast)`）は**統合音声1本に置換**＝automation の `音声化待ち` 波及（`notion_status_automation.py` L249）を「日次ダイジェスト単位」に寄せる必要あり＝ここはサクと要相談。

### 媒体全体（確定事項・再掲）
- 毎日：デイリー記事＋音声（高=厚く＋低=一行・国際保健毎日）。週末：ウィークリー（note有料）＝毎日の蓄積を再編集（既存 `weekly_policy_report.py`）。無料リーチ／有料フロー継続＝CrossHealthサブスク戦略と一致。
